## Day 1 — Backend API болон өгөгдлийн загвар

**Зорилго:** Өгөгдлийн загварчлал (data modeling), REST API зохиомж, бизнес логик бичих чадварыг үнэлэх.

### Даалгавар

Хялбаршуулсан "Даатгалын гэрээ удирдах" API-г боловсруулна. Хоёр үндсэн entity байна:

#### Customer (даатгуулагч)
- `id`
- `first_name`
- `last_name`
- `register_number` (регистрийн дугаар)
- `birth_date`
- `phone`

#### Policy (даатгалын гэрээ)
- `id`
- `customer_id`
- `policy_type` (`car` / `home` / `travel`)
- `base_amount` (даатгалын үнэлгээний дүн)
- `start_date`
- `end_date`
- `premium` (автоматаар тооцоологдож оноогдоно)
- `status` (`active` / `expired` / `cancelled`)

### Шаардагдах endpoint-ууд

```
POST   /customers              # Шинэ даатгуулагч бүртгэх
GET    /customers              # Даатгуулагчдыг жагсаах
GET    /customers/{id}         # Нэг даатгуулагчийг гэрээнүүдтэй нь хамт авах

POST   /policies               # Шинэ гэрээ үүсгэх (premium автоматаар тооцоологдоно)
GET    /policies               # Бүх гэрээ (шүүлт: ?customer_id, ?status)
GET    /policies/{id}          # Нэг гэрээний дэлгэрэнгүй
PATCH  /policies/{id}/cancel   # Гэрээг цуцлах
```

### Premium тооцоолох бизнес логик

```
base_premium = base_amount × 0.02    # үндсэн тариф 2%

Насанд суурилсан коэффициент (birth_date-аас тооцоолно):
  25-аас доош нас → × 1.5
  25–50 нас       → × 1.0
  50-аас дээш нас → × 1.3

Төрөлд суурилсан коэффициент:
  car    → × 1.2
  home   → × 1.0
  travel → × 0.8

premium = base_premium × age_factor × type_factor
```

### Шалгалт (validation)-ын шаардлага

- Регистрийн дугаар нь 10 оронтой байх (эхний 2 нь үсэг, үлдсэн 8 нь тоо)
- `end_date` нь заавал `start_date`-аас хойш байх ёстой
- `base_amount` эерэг тоо байх
- Буруу өгөгдөл ирсэн тохиолдолд тохирох HTTP статус код (400, 404 гэх мэт) буцаах

### Хүргэх үр дүн

- Ажиллагаатай API
- Суулгах болон ажиллуулах зааврыг `README.md` файлд тусгасан байх
- Хамгийн багадаа **утга учиртай 3 commit**

## Day 2 — Frontend интерфэйс

**Зорилго:** Хэрэглэгчийн интерфэйс (UI) бүтээх, API-тай холболт хийх, хэрэглэгчийн туршлагыг (UX) бодох чадварыг үнэлэх.

### Даалгавар

Day 1-д бичсэн API-г ашиглан энгийн веб интерфэйсийг хөгжүүлнэ.

### Функциональ шаардлага

1. **Даатгуулагчийн хуудас**
   - Шинээр бүртгэх маягт
   - Бүртгэлтэй даатгуулагчдын жагсаалт

2. **Гэрээний хуудас**
   - Шинэ гэрээ үүсгэх маягт (даатгуулагч сонгох dropdown-тэй)
   - Premium-ийг хадгалахаас өмнө **шууд (real-time) урьдчилан харуулах**

3. **Даатгуулагчийн дэлгэрэнгүй**
   - Даатгуулагч дээр дарахад түүний бүх гэрээ харагдана

4. **Цуцлах үйлдэл**
   - Идэвхтэй (active) гэрээг товчоор цуцлах боломжтой байх

### Техникийн шаардлага

- React, Vue эсвэл цэвэр HTML/CSS/JS — сонголтоор
- **Цэгц сайтай CSS** ашиглах (бүгд inline биш, бүтэц нь ойлгомжтой)
- Ачаалах үеийн төлөв (loading state) болон алдааны мэдэгдлийг харуулах
- Форм шалгалт (form validation)-ыг frontend талд бас хийсэн байх

### Хүргэх үр дүн

- Код нь нэг repository дотор эсвэл `frontend/` хавтас дотор байрлана
- README-д frontend-ийг ажиллуулах зааврыг нэмэх
- Хамгийн багадаа **4 commit** (Day 1+2 нийлээд 7-оос доошгүй commit байх)

## Day 3 — Өгөгдлийн шинжилгээ, баримтжуулалт, эргэцүүлэл

**Зорилго:** Шинэ өгөгдлийг шинжлэх, баримт бичиг үйлдэх, өөрийн ажлыг үнэлэх чадварыг хэмжих.

### 3.1 Өгөгдлийн шинжилгээний script

Бид танд `claims.csv` нэртэй файл (ойролцоогоор 500 мөр нэхэмжлэлийн өгөгдөл) өгнө:

```
claim_id, policy_id, policy_type, claim_date, claim_amount, status
```

Python эсвэл JavaScript ашиглан дараах асуултуудад хариулах script бичнэ:

1. Нийт хэдэн нэхэмжлэл байна вэ? Үүний хэдэн хувь нь `approved` төлөвтэй вэ?
2. Policy type тус бүрийн дундаж нэхэмжлэлийн дүн хэд вэ?
3. Хамгийн их нэхэмжлэл гарсан 3 сар аль нь вэ?
4. Policy-ийн дүн болон нэхэмжлэлийн дүнгийн харьцаа (loss ratio) policy type тус бүрээр ямар байна вэ?

Үр дүнг терминалд цэвэрхэн хэвлэх эсвэл `report.json` файлд хадгална.

### 3.2 README.md баримт бичиг

Төслийн бүрэн README-г бичих. Дараах хэсгүүдийг багтаасан байх ёстой:

- Төслийн танилцуулга (2–3 өгүүлбэр)
- Архитектур (энгийн диаграмм, Mermaid ашиглавал нэмэлт оноотой)
- Суулгах болон ажиллуулах заавар
- API endpoint-уудын жагсаалт жишээ request/response-ийн хамт
- Техникийн сонголтын үндэслэл (Яагаад энэ хэл, framework-ийг сонгосон бэ?)
- Тулгарсан бэрхшээл болон шийдсэн арга барил

### 3.3 Өөрийгөө эргэцүүлэх

`REFLECTION.md` нэртэй тусдаа файл үүсгэж, дараах асуултуудад үнэнчээр хариулна (1 хуудас):

1. Энэхүү 3 хоногийн хугацаанд юу шинийг сурсан бэ?
2. Хамгийн хэцүү хэсэг нь юу байсан, яаж даван туулсан бэ?
3. Хэрэв дахин эхнээс нь хийх боломжтой бол юуг өөрөөр хийх байсан бэ?
4. Өгсөн хугацаандаа дуусгаж амжаагүй зүйлс, мөн тэдгээрийг цаашид хэрхэн сайжруулах санаа
